# CI Workflow Design

**Context**

The repository now has three testing layers:

1. `vitest` unit tests
2. Playwright source-level e2e tests
3. Playwright packed-artifact smoke tests

The existing GitHub Actions workflow runs install, unit tests, coverage, build, and deployment in a single job. It does not yet account for Playwright browser setup or differentiated trigger strategy for smoke coverage.

**Goal**

Refactor CI so that pull requests validate the main source-level test path, while `main` branch pushes additionally validate the packed npm artifact before deployment.

**Decision**

Keep a single workflow file, but split it into multiple jobs:

1. `verify`
2. `smoke`
3. `deploy`

**Why This Approach**

1. Keeps repository complexity low by avoiding multiple workflows.
2. Separates responsibilities clearly inside the same workflow.
3. Prevents smoke checks from slowing down every pull request.
4. Ensures docs deployment only happens after all required validation passes on `main`.

**Trigger Strategy**

1. Pull requests to `main`
   - run `verify`
2. Pushes to `main`
   - run `verify`
   - run `smoke`
   - run `deploy`

**Job Responsibilities**

1. `verify`
   - install pnpm and Node in a standard way
   - install dependencies
   - install Playwright Chromium
   - run:
     - `pnpm test:run`
     - `pnpm test:e2e`
     - `pnpm build`
   - optionally generate coverage on `main`
2. `smoke`
   - run only on `push` to `main`
   - install dependencies and Chromium
   - run `pnpm test:smoke`
3. `deploy`
   - run only on `push` to `main`
   - depend on successful `verify` and `smoke`
   - deploy docs
   - upload coverage if produced

**Out of Scope**

1. Release workflow changes
2. Matrix testing across Node versions or browsers
3. Parallel multi-workflow refactor

**Stability Rules**

1. Use `actions/setup-node` and `pnpm/action-setup` instead of global npm installation.
2. Install only Chromium, not all Playwright browsers.
3. Keep `smoke` off the PR path unless future risk or requirements change.
